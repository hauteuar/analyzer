AI Context Blotting & Compression

Intelligent context management that reduces token usage by 60-80% while preserving semantic meaning and conversation quality.

üóúÔ∏è What is AI Blotting?

AI Blotting is an intelligent compression technique that summarizes and condenses conversation history, documents, and context before sending to LLMs. It preserves critical information while dramatically reducing token costs and improving response times.

üéØ Blotting Strategies

  • Conversation Blot: Summarize chat history into key points
  • Document Blot: Extract relevant sections from large docs
  • Query-Aware Blot: Keep context relevant to current query
  • Temporal Blot: Prioritize recent over old context
  • Entity Blot: Preserve named entities and relationships

‚öôÔ∏è Compression Levels

  • Light (20%): Remove filler, keep structure
  • Medium (50%): Summarize paragraphs to sentences
  • Heavy (80%): Extract key facts only
  • Adaptive: Auto-adjust based on token budget
  • Selective: User-defined preservation rules

üìà Benefits

  • Cost Reduction: 60-80% fewer tokens = lower API costs
  • Faster Response: Less context = faster LLM processing
  • Longer Memory: Fit more history in context window
  • Better Focus: Remove noise, improve accuracy
  • Compliance: Control what data reaches LLM

üõ†Ô∏è Configuration

  • Token Budget: Set max tokens per request
  • Preserve Rules: Always keep certain entities
  • Ignore Patterns: Skip specified content types
  • Priority Weights: Score context by importance
  • Cache Strategy: Reuse blotted summaries
Blotting Node Configuration (Visual Builder)
# Context Blotting Node - Visual Configuration

node_type: "context_blot"
name: "Compress Conversation History"

input:
  source: "conversation_history"
  max_messages: 100
  include_system: false

blotting:
  strategy: "query_aware"           # Optimize for current query
  compression_level: "adaptive"     # Auto-adjust to budget
  target_tokens: 4000              # Target output size
  preserve_recent: 5               # Keep last 5 messages verbatim

preservation_rules:
  entities:                         # Always preserve these
    - "account_numbers"
    - "trade_ids"
    - "violation_codes"
  keywords:
    - "compliance"
    - "violation"
    - "deadline"

output:
  format: "structured"
  sections:
    - "summary"                      # High-level summary
    - "key_facts"                    # Extracted facts
    - "recent_context"               # Recent messages
    - "entities"                     # Preserved entities

üìã What is SKILL.md?

SKILL.md files are structured markdown documents that define AI agent capabilities. They include instructions, examples, constraints, and tool definitions that are injected into the agent's context when needed. Think of them as "instruction manuals" for AI agents.

# SKILL.md - Compliance Violation Checker
# Location: /skills/compliance/violations.skill.md

---
name: "Compliance Violation Checker"
version: "2.1.0"
category: "compliance"
triggers:
  - "violation"
  - "day trade"
  - "90 day"
  - "good faith"
  - "compliance check"
---

## Description
You are a compliance specialist for BOFA Wealth Management. Your role is to 
check for trading violations including day trade violations, 90-day rule 
violations, and good faith violations.

## Instructions
1. When asked about violations, ALWAYS extract the account ID first
2. Query the compliance database using the DB2 MCP
3. Check for ALL violation types unless user specifies one
4. Format results with severity levels (HIGH/MEDIUM/LOW)
5. Include violation codes and remediation steps

## Available Tools
```yaml
tools:
  - name: "db2_compliance_query"
    mcp: "db2-compliance"
    action: "executeQuery"
    description: "Query compliance violations from DB2"
    
  - name: "check_day_trade"
    mcp: "db2-compliance"
    action: "callProcedure"
    procedure: "SP_CHECK_DAY_TRADE"
    
  - name: "check_90_day"
    mcp: "db2-compliance"
    action: "callProcedure"
    procedure: "SP_CHECK_90_DAY_RULE"
```

## Examples

### Example 1: Check all violations
User: "Check violations for account 12345"
Agent: 
1. Extract account_id: 12345
2. Call check_day_trade(account_id=12345)
3. Call check_90_day(account_id=12345)
4. Format and return results

### Example 2: Specific violation type
User: "Any day trade violations today?"
Agent:
1. Extract date: today
2. Call check_day_trade(date=today)
3. Return day trade violations only

## Constraints
- Never expose raw SQL queries to users
- Always mask full account numbers in responses (show last 4 digits)
- Escalate HIGH severity violations to compliance team
- Response must include violation code for audit trail
